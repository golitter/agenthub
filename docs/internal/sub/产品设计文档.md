## 1. 项目背景

### 1.1 行业背景

编码 Agent（Claude Code、OpenCode、Codex 等）已能自主读写文件、执行命令、提交代码，**能力早已溢出编码本身——同样覆盖内容创作、文档处理、数据分析、日常办公等开发与生活场景**。但每个 Agent 仍是孤岛：**上下文不共享、产出分散、过程黑盒、接入成本高**。与此同时，「像用飞书 / 微信一样与 AI 协作」的对话式范式正成为主流，已被验证为非技术用户也能上手的最低门槛。

### 1.2 课题背景

课程课题「AgentHub - 多 Agent 协作平台」要求构建**以 IM 聊天为核心范式**的多 Agent 协作平台——每个 Agent 就是一个「聊天对象」，用户通过建群、@、发消息与之协作。

考察维度与权重：

| 维度 | 权重 | 评判要点 |
|------|------|----------|
| AI 协作能力 | 30% | 沉淀 Spec / Skill / Rules 等协作规范 |
| 功能完整度 | 25% | IM 流畅、多 Agent 调度跑通 |
| 生成效果质量 | 20% | 聊天 UI 与产物预览效果 |
| 代码理解度 | 15% | 能否解释架构选型与核心逻辑 |
| 创新与产品感 | 10% | 超预期功能与产品设计方案 |

## 2. 目标与指标

### 2.1 产品目标

按 P0 必做 / P1 应做 / P2 可选 三档划分：

| 优先级 | 目标 | 描述 |
|--------|------|------|
| P0 | IM 核心体验跑通 | 新建会话、单聊、流式输出、上下文连续、历史回放 |
| P0 | 多 Agent 调度跑通 | 群聊 + Orchestrator 自动拆解、分派、聚合 |
| P0 | 至少接入 2 个主流 Agent | Claude Code + Codex / OpenCode（实际接入 3 个 + Orchestrator） |
| P1 | 产物内联预览 | 代码 Diff、网页 HTML 卡片、图片在聊天流中直接预览 |
| P1 | 人工介入点 | Orchestrator 规划完成后必须经用户审查才执行 |
| P2 | 工作区隔离 | 每个 Agent 独立 git worktree，互不污染 |
| P2 | 可自建 Agent | 通过 SOUL.md 身份文档 + 技能定义塑造个性化 Agent |

### 2.2 非目标（明确不做）

为守住「务实优先、不过度设计」的纪律，明确以下为非目标：

- ❌ **不做通用任务调度引擎**——本课题以「会话内的即时编排」为目标，不解决跨节点 / 跨进程的分布式任务调度问题。
- ❌ **不做通用资产 / 数据血缘系统**——产物仅做引用 + 预览，不做版本 / 依赖 / 血缘追踪。
- ❌ **不做多租户 / 权限体系**——课题明确单人使用，搭一整层用户与权限基础设施性价比低。
- ❌ **不做实时协同编辑**——产品定位是「对话式协作」而非「多人实时共编」，单向流式推送足够覆盖。
- ❌ **不让后端变成第二个执行引擎**——后端只承担「状态权威 + 中继 + 持久化」，Agent 执行统一收敛到 Agent 端，避免双执行入口带来的状态分裂。

## 3. 用户与场景

### 3.1 目标用户

AgentHub 是单人系统（单用户），但「用户角色」按使用心智拆分为三个画像，用于指导交互设计：

| 画像 | 特征 | 核心诉求 | AgentHub 对应能力 |
|------|------|----------|-------------------|
| **开发者（主画像）** | 熟悉 Git / 命令行，但同时维护多个项目，希望并行委派任务 | 把琐碎编码任务批量交给不同 Agent，自己只做决策 | 多会话并行、群聊协作、Diff 一键应用 |
| **项目管理者** | 不直接写代码，但需要审核方案、跟踪进度 | 看得懂 Agent 在做什么、能随时叫停 / 修正 | 规划审查卡、运行时状态卡、执行总结卡 |
| **轻度用户 / 未来扩展** | 不熟命令行，想用对话生成网页 / 文档 | 零门槛、所见即所得 | 产物内联预览、网页卡片、一键部署 |

### 3.2 核心使用场景

**场景 A：单聊——明确任务直达**
让 Claude Code「写一个带表单校验的登录组件」，流式输出代码并生成 Diff 卡，一键应用进工作区。

**场景 B：群聊——多 Agent 协作**
建含 Orchestrator + Claude Code + Codex 的群聊，发「做一个产品落地页（Hero / 特性 / Footer）」。Orchestrator 自动拆成「结构 + 样式」分派；审查批准后，两个子 Agent 像群成员一样依次产出，Orchestrator 最后聚合汇报。

**场景 C：跨 Agent 记忆**
@Codex「补一下样式」——它执行前自动看到 Claude Code 刚才的发言，基于已有结构续写，而非从零开始。

**场景 D：断线恢复**
编排中途刷新页面：已完成的消息、规划卡、产物卡全部还在，正在流式输出的任务断点续传、不重不丢。

**场景 E：个性化 Agent（SOUL.md）**
给 Codex 写一份身份文档（如「严谨的 TypeScript 工程师，函数必须有类型注解」），此后每次执行自动注入，产出风格稳定一致。

**场景 F：技能供给（SkillsHub）**
上传自定义技能包（如「html-card」）入库并导入 Claude Code，让它把结果渲染成可预览网页卡片。

### 3.3 典型用户旅程（端到端）

```
登录 → 看到会话列表（左侧）
  │
  ├─ 新建单聊：选 Agent → 发消息 → 流式输出 → 应用产物
  │
  └─ 新建群聊：选多个 Agent（含 Orchestrator）→ 发消息
        → Orchestrator 规划 → [审查卡] 用户批准
        → 子 Agent 依次执行（运行时状态卡实时更新）
        → 产物内联（Diff / HTML / 图片）
        → Orchestrator 聚合总结
        → 用户应用 / 部署 / 二次迭代
```

## 4. 产品范围

### 4.1 范围边界

- **是**：一个本地可运行、Demo 可演示的多 Agent 协作平台。
- **不是**：一个生产级 SaaS（无多租户、无计费、无 SLA 保障）。
- **权限策略**：编码 Agent 的工具调用权限沿用 Agent 自身能力（平台层不做额外沙箱）；AgentHub 在平台层保留**审查入口**——关键产物应用前需用户确认（如 Diff 卡），Orchestrator 规划需经审查 + 人工介入。

### 4.2 功能范围矩阵

按课题 6 大核心功能 × 优先级（P0 必做 / P1 应做 / P2 可选）划分：

| 功能域 | P0 | P1 | P2 |
|--------|----|----|----|
| **IM 聊天交互** | 会话列表、单聊、流式输出、历史回放、断线重连 | 置顶、通讯录分组、退出群聊 | 消息搜索、消息操作（回复 / 引用） |
| **Orchestrator** | 自动拆解、分派、聚合 | 规划审查（人工介入）、重规划 | 失败降级、代码冲突处理 |
| **多 Agent 接入** | Claude Code、Codex、OpenCode 适配层 | Orchestrator 编排器 | 自建 Agent（SOUL.md + 技能） |
| **产物预览编辑** | Diff 卡片、HTML 网页卡片、图片卡片 | 全屏预览、一键应用 Diff | Diff 版本历史、对话式局部修改 |
| **部署发布** | — | 预览 URL、源码打包下载 | 静态站点部署、容器化部署 |
| **工作区隔离** | git worktree 隔离 | shared 目录共享 | — |
| **技能体系** | 内置技能（render / taskctl） | SkillsHub 上传 / 导入 / 移除 | — |
| **多端支持** | Web 端（主力） | — | 桌面端、移动端 |

> 本期交付覆盖全部 P0 + 大部分 P1；P2 中「部署发布」与「多端」为远期，详见第 11 章。

## 5. 功能需求

本章按功能域展开需求条目（FR = Functional Requirement），每条标注优先级与验收要点。

### 5.1 IM 聊天交互

| 编号 | 需求 | 优先级 | 验收要点 |
|------|------|--------|----------|
| FR-IM-01 | 会话列表：左侧展示所有会话，按「置顶优先 + 最近活跃」排序 | P0 | 置顶会话带 Pin 图标且排在最前 |
| FR-IM-02 | 新建会话：选择单个 Agent（单聊）或多个 Agent（群聊） | P0 | 群聊必须可包含 Orchestrator |
| FR-IM-03 | 单聊流式输出：消息逐字流式显示，带光标 | P0 | 流式过程视觉流畅、无明显卡顿 |
| FR-IM-04 | 上下文连续：会话历史作为上下文传递给 Agent | P0 | 支持多轮迭代修改 |
| FR-IM-05 | 历史回放：刷新 / 重进会话，历史消息完整 | P0 | 含规划卡、运行时卡、产物卡 |
| FR-IM-06 | 断线重连：连接中断后自动续传，不重不丢 | P0 | 用户感知不到中断 |
| FR-IM-07 | 置顶会话：侧栏「置顶 / 取消置顶」 | P1 | 置顶状态在刷新后保留 |
| FR-IM-08 | 通讯录分组：会话可分到自定义分组 | P1 | 分组可折叠、未分组区域独立 |
| FR-IM-09 | 退出群聊：彻底删除会话及工作区 | P1 | 二次确认；消息与工作区文件一并清除 |
| FR-IM-10 | 消息操作：回复 / 引用 / 重新生成 / 复制 | P2 | 仅复制可用，回复 / 引用 / 重新生成未实现 |

### 5.2 Orchestrator（主 Agent 协调器）

| 编号 | 需求 | 优先级 | 验收要点 |
|------|------|--------|----------|
| FR-OR-01 | 意图理解：自动理解用户复杂任务 | P0 | 产出可读的执行规划 |
| FR-OR-02 | 任务拆解：拆为子任务并分派给合适子 Agent | P0 | 子任务以波次（并行 / 串行）可视化呈现 |
| FR-OR-03 | 结果聚合：子 Agent 完成后聚合并在聊天流汇报 | P0 | 出现最终总结卡 |
| FR-OR-04 | 规划审查：规划产出后**强制暂停**等待用户批准 | P1 | 每次执行前必审、不可绕过 |
| FR-OR-05 | 讨论 / 修改：审查时可提意见触发重规划 | P1 | 反馈被纳入新一轮规划 |
| FR-OR-06 | 跨 Agent 记忆：子 Agent 执行前看到其他 Agent 发言 | P1 | 后执行的 Agent 不重复已完成的工作 |
| FR-OR-07 | 失败降级 / 冲突处理 | P2 | 检测到冲突后给出回滚建议 |

### 5.3 多 Agent 接入

| 编号 | 需求 | 优先级 | 验收要点 |
|------|------|--------|----------|
| FR-AD-01 | Claude Code 接入 | P0 | 可在 IM 中调用并流式输出 |
| FR-AD-02 | OpenCode 接入 | P0 | 可在 IM 中调用并流式输出 |
| FR-AD-03 | Codex 接入 | P1 | 可在 IM 中调用并流式输出 |
| FR-AD-04 | Orchestrator 接入（调度层） | P0 | 可在群聊中自动编排 |
| FR-AD-05 | 统一 Agent 抽象 | P0 | 新增 Agent 无需改动前端 |
| FR-AD-06 | 自建 Agent：SOUL.md 身份 + 技能集 | P2 | 编辑 ≤ 300 字身份 |

### 5.4 产物预览与编辑

| 编号 | 需求 | 优先级 | 验收要点 |
|------|------|--------|----------|
| FR-AR-01 | Diff 卡片：多文件 tab 切换查看 | P0 | 可一键应用 |
| FR-AR-02 | HTML 网页卡片：聊天流内联预览 | P0 | 隔离运行、不影响主界面 |
| FR-AR-03 | 图片卡片：产物图片展示 | P0 | 产物图片按 session 路径内联渲染；加载失败显示占位、不阻塞 |
| FR-AR-04 | 工具进度卡：实时显示命令输出 | P0 | 实时刷新、不卡 UI |
| FR-AR-05 | 全屏预览 / 代码编辑器 | P1 | 点击卡片展开 |
| FR-AR-06 | 一键应用 Diff：写入工作区 | P1 | 应用前可二次确认 |
| FR-AR-07 | Diff 版本历史、对话式局部修改 | P2 | 对话式局部修改：编码 Agent（单聊）已支持、Orchestrator（群聊）不支持；Diff 版本历史：未实现 |

### 5.5 消息路由（群聊 @ 机制）

| 编号 | 需求 | 优先级 | 验收要点 |
|------|------|--------|----------|
| FR-RT-01 | `@某Agent`：命中唯一非 Orchestrator 时直达该 Agent | P0 | 绕过 Orchestrator，更快 |
| FR-RT-02 | 群聊无 `@`：由 Orchestrator 自动编排 | P0 | 无 @ 时触发 Orchestrator 规划（出现规划卡），不直达任意子 Agent |
| FR-RT-03 | `@all` / 多个 `@` / `@orchestrator`：交 Orchestrator | P1 | 多目标协作 |
| FR-RT-04 | 单聊无 `@`：当前 Agent 直达 | P0 | 保持单聊语义 |
| FR-RT-05 | `@未知Agent`：前端提示 / 后端拒绝 | P1 | 不静默派错 |
| FR-RT-06 | Mention 仅在消息开头生效 | P1 | 避免误伤正文引用 |

> 路由决策是群聊体验的核心机制，FR-RT-01/02 与 FR-OR 的拆解/分派同等基础，因此标记为 P0；FR-RT-03 是「多目标协作」增强能力，归 P1。

### 5.6 技能体系（SkillsHub）

| 编号 | 需求 | 优先级 | 验收要点 |

| FR-SK-02 | 上传技能：上传压缩包 → 校验 → 确认入库 | P1 | 校验大小 / 文件数 / 路径合法性 |

## 6. 信息架构

### 6.1 全局导航结构

```
AgentHub（Web SPA）
├── 左侧 Icon Sidebar（主导航）
│   ├── 💬 聊天（ImPage）        ← 默认主页，会话列表 + 聊天区
│   ├── 👥 通讯录（ContactsPage） ← 会话分组管理
│   ├── 🧩 技能（SkillsHubPage）  ← 技能仓库
│   ├── 📊 管理（Admin）          ← Agent / 系统管理
│   └── ⚙ 设置
│
├── 聊天页三栏布局
│   ├── 左：ConversationList（会话列表 + 搜索）
│   ├── 中：ChatArea（消息流 + 输入框）
│   └── 右：RightSidebar（会话信息：置顶 / 退群 / 成员 / SOUL.md）
│
└── Agent 详情页（AgentProfilePage）
    ├── 基础信息（头像 / 名称 / Agent 类型）
    ├── SOUL.md 编辑（≤ 300 字身份文档）
    └── Skills 区域（内置 + 外置技能，导入 / 移除）
```

### 6.2 信息层级

| 层级 | 实体 | 说明 |
|------|------|------|
| L1 | 导航 Tab | 聊天 / 通讯录 / 技能 / 管理 |
| L2 | Task（会话） | 一个聊天窗口，含若干 Session |
| L3 | Session | 一个 Agent 的运行上下文 |
| L4 | Message | 消息（含结构化 block） |
| L5 | MessageBlock | 消息内的渲染单元（text / diff / html / plan / runtime_status...） |

### 6.3 会话列表组织规则

- 排序：置顶优先，再按最近活跃时间倒序。
- 分组：未归入任何自定义分组的会话，统一进入「未分组」区域。
- 搜索：按会话标题 / Agent 名称模糊匹配。

## 7. 流程设计

### 7.1 单聊流式输出流程（端到端）

```
用户输入消息
  → 前端发起任务
  → 后端创建消息 + 启动流式通道
  → 调用 Agent 端执行
  → Agent 端按事件流式回传（文本 / 工具 / 产物）
  → 后端中转：实时推送前端 + 同时持久化
  → 前端按事件类型增量渲染卡片
```

### 7.2 群聊编排流程（Orchestrator）

```
用户在群聊发消息（无 @）
  → 路由判定：交给 Orchestrator
  → Orchestrator 推理规划 → 产出执行方案（波次 + 子任务）
  → ⏸ 暂停等待审查
       ├── 用户批准 → 进入分派与执行
       └── 用户讨论 / 修改 → 携反馈回到规划阶段
  → 子 Agent 按波次依次执行（波次内并行、波次间串行）
  → 子任务产出聚合 → 最终总结卡汇报给用户
```

### 7.3 消息路由决策流程

```
用户在群聊发送消息
  │
  ├─ 消息开头 @唯一非 Orchestrator Agent ──▶ 直达该 Agent
  ├─ 群聊无 @ 且存在 Orchestrator         ──▶ 交 Orchestrator 自动编排
  ├─ @all / 多个 @ / @orchestrator         ──▶ 交 Orchestrator
  └─ @未知 Agent                           ──▶ 前端即时提示、不静默派错

用户在单聊发送消息
  │
  └─ 单聊无 @ ──▶ 当前 Agent 直达
```

### 7.4 规划审查流程（人工介入）

```
Orchestrator 产出规划
  → 暂停运行、聊天流原位展示规划卡（含波次 / 子任务 / 反馈框）
  → 等待用户决策（默认超时 10 分钟）
       ├── 批准 ────▶ 进入分派执行
       └── 讨论 / 修改 ▶ 携反馈回到规划阶段重规划
  → 决策后继续运行；等待期间持续保活，避免长连接被代理断开
```

边界处理：超时后自动批准；进程异常时下次重进可恢复至等待状态。

### 7.5 跨 Agent 记忆注入流程

```
子 Agent A 即将在群聊中执行
  → 后端查询「当前会话中其他 Agent 最近的一段发言」（每条截断保护）
  → 作为上下文注入 A 的请求
  → A 的输出由此建立在前序 Agent 的工作之上，而非从零开始

单聊中此流程自然为空（只有 1 个 Agent），不影响单聊体验。
波次串行处理天然保证波次间的窗口隔离。
```

### 7.6 断线重连流程

```
连接断开 → 前端自动重连（指数退避）
  → 后端先回放断线期间产生的历史事件
  → 回放完毕后切换为实时推送
  → 事件顺序严格递增，不丢不重
```

## 8. 交互设计

### 8.1 流式渲染体验目标（核心体验）

流式输出场景下，每个字到达的间隔很短，渲染策略要保证人眼流畅、长会话不卡：

1. **节流渲染**：把高频到达的文字按帧批量刷出，避免逐字重渲染。
2. **长列表虚拟化**：消息数较多时，仅渲染可视区域，避免 DOM 节点膨胀。
3. **流式光标**：流式输出时显示闪烁光标，提示生成中。
4. **智能滚动**：用户在底部时自动跟随；用户主动上滚时停止跟随，并展示「回到底部」按钮。
5. **断线去重**：断线重连后回放的历史可能与已接收内容重叠，前端按消息身份跳过重复，只追加新内容。

### 8.2 消息气泡设计

- **User**：右对齐，柔和主色背景。
- **Agent**：左对齐，Agent 头像 + **颜色标识条**（不同 Agent 不同颜色）。
- **System**：居中，灰色小字。
- **宽度自适应**：纯文本较窄，结构化内容（规划 / Diff）更宽。
- **长消息折叠**：超过一定长度时预览展示，点击放大查看完整内容。

### 8.3 卡片类型与交互

不同类型的产物 / 状态在聊天流中映射为不同卡片，**完整的卡片家族清单与组件命名见 [9.5 消息卡片家族](#95-消息卡片家族)**，本表仅按**交互模式**做归类速览：

| 交互模式 | 覆盖卡片 | 典型行为 |
|---------|---------|---------|
| 只读展示 | 文本 / RuntimeStatus / ImageCard / AttachmentCard / FinalSummaryCard / ToolCard | 仅渲染、不接收用户输入 |
| 内联 / 全屏预览 | HtmlCard / PreviewCard | 内联缩略 → 点击放大或全屏 |
| 多文件切换 + 应用 | DiffCard | 文件 Tab + 增删高亮 + [应用] |
| 用户决策 | PlanReviewCard / AskAgentCard | 批准 / 讨论 / 回复 |
| 折叠 / 展开 | CoordChannel 协作通道 | 默认折叠，按需展开 |
| 错误反馈 + 重试 | TaskFailureCard | 显示错误 + [重试] |

### 8.4 关键交互细节

- **@ 选择**：输入 `@` 弹出群成员列表，选择后插入 `@名称 `；未知成员即时提示。
- **会话切换无感**：左栏点击会话即时切换，历史消息快速回放。
- **规划审查卡**：在聊天流中原位出现，批准后无缝继续，讨论时新消息走审查通道而非新建任务。
- **产物应用**：Diff 卡片「应用」按钮直接写入工作区，无需切到终端。
- **退出群聊二次确认**：弹出确认框，明确提示「将彻底删除所有消息和工作区数据」。

### 8.5 可访问性与反馈

- 所有异步操作有明确状态：空闲 → 加载中 → 流式中 → 工具运行 → 完成 / 错误。
- 流式过程中工具调用显示加载指示器，工具结果即时展示。
- 错误以独立卡片明确呈现，不静默失败。

### 8.6 视觉风格规范

AgentHub 的视觉定位是 **Dark Utilitarian（深色 · 平坦 · 克制 · 密集）**——它不是聊天机器人，而是「AI Runtime Workspace / 精密控制台」，气质对标 Linear 的克制暗色面板、Cursor 的 IDE 氛围、Slack 的对话节奏感。

**主题**：项目实现暗 / 亮双主题，以暗色为主基调（画布 `#0A0B0E` → 卡片 `#1A1D24` 的 5 级灰阶分层），通过 CSS 变量（亮 / 暗两套）实现双主题切换，亮色架构相同。

**风格**遵循三条原则：**层级靠色阶、不靠阴影**（深度由背景色差表达，几乎不用 `box-shadow`）；**色彩靠功能、不靠装饰**（全局仅 1 个品牌色 `#6366F1` + Agent 标识色用于身份 + 3 个语义色，禁用渐变与毛玻璃）；**动效靠存在、不靠表现**（仅 120–300ms 状态过渡，只动 `transform` / `opacity`）。群聊中 Agent 靠**左侧 3px 标识色条 + 圆角方形头像**区分身份（Claude `#DA7756` / OpenCode `#10B981` / Orchestrator `#EAB308` / Codex `#6366F1`），字体统一 Geist（UI）+ Geist Mono（代码），圆角基准 10px、间距基准 4px。

## 9. 原型设计

> 本章按前端实际视图结构组织，每个页面的关键形态以「文字 + ASCII 线框」呈现。
>
> **原型生成路径**：每张原型都通过「设计者 ↔ AI 编程助手（aicoding）」的多轮交互探讨产出 — 先由设计者描述页面意图、布局骨架与交互要求，aicoding 生成首版 HTML 原型（含 design token、栅格、状态切换、动效占位）；设计者再就视觉细节、文案、边界态反复校准，迭代到符合产品意图后，沉淀为「**开发原型 HTML**」，作为前端实现的视觉与交互基线。本章的 ASCII 线框是这些 HTML 原型的「概要投影」，便于快速通读与评审；真实交互、间距、配色以 HTML 原型为准。
>
> **迭代节奏**：① 设计者口述需求 → ② aicoding 出首版 HTML → ③ 设计者逐项指出偏差（间距 / 文案 / 状态 / 动效）→ ④ aicoding 改版 → ⑤ 多轮往返直至通过 → ⑥ 固化为开发原型 HTML。后续 UI 实现以该 HTML 为像素级参考，按其拆解组件与状态机。
>
> 前端共 **4 个主视图**（聊天 / 通讯录 / 技能库 / 管理后台）+ **1 个独立路由**（Agent 详情页），由最左侧 `IconSidebar` 统一切换。

## 10. 非功能需求

### 10.1 性能

| 维度 | 要求 |
|------|------|
| 首 token 延迟 | P95 ≤ 2s |
| 流式渲染帧率 | 稳定 60fps |
| 长会话渲染 | 单会话 50 条以上消息不卡顿 |
| 高吞吐事件 | 高频工具输出（如依赖安装）下不内存溢出、不堵塞流式通道 |

### 10.2 可靠性

| 维度 | 要求 |
|------|------|
| 断线重连 | 不丢不重 |
| 历史回放 | 刷新后所有卡片完整可见 |
| 关键终态事件 | 绝不丢失 |
| 进程崩溃 | 等待审查中的任务下次重进可恢复至等待状态 |

### 10.3 安全

| 维度 | 要求 |
|------|------|
| Agent 端隔离 | 前端不直连 Agent 端，必须经后端中转与认证 |
| 技能包安全 | 上传压缩包须校验总大小、文件数、路径合法性，防止恶意构造 |
| 工作区路径 | 不向客户端暴露工作区绝对路径 |
| 输入校验 | SOUL.md ≤ 300 字、技能命名不冲突，前后端双重校验 |

### 10.4 可扩展性

- **Agent 可插拔**：新增 Agent 接入对前端零改动。
- **事件类型可演进**：新增事件类型不破坏既有解析。
- **状态机可演进**：新增状态不破坏既有转换。

### 10.5 可维护性

- **三层职责清晰**：前端消费 / 后端中继 / Agent 端执行各司其职。
- **跨端类型单一来源**：契约层统一三端类型定义，避免漂移。

### 10.6 部署

- **一键起三端**：单条命令启动前端 / 后端 / Agent 端，开发期支持热重载。
- **容器化**：提供容器化部署方案，三端隔离。
- **可观测**：三端运行日志分文件输出，单条命令查看服务状态。

### 10.7 兼容性

- **事件版本兼容**：事件结构升级后，老版本回放仍可解析。
- **接口向后兼容**：后端响应新增字段不破坏旧前端。

### 11.2 潜在风险

> 概率分级：🟢 低 / 🟡 中 / 🔴 高；影响分级：① 轻微 / ② 中等 / ③ 严重。

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| Orchestrator 误判（派错 Agent / 拆解不当） | 🟡 | ③ | 规划审查强制人工介入；@ 直达绕过编排减少误判面 |
| Agent 异常退出 / 超时 | 🟡 | ② | 自动重试 + 异常卡片明确告知用户 |
| 高频工具输出（如依赖安装、批量测试） | 🟢 | ② | 流量整形 + 关键事件优先保护 |
| 多 Agent 同名 / 同类型导致路由混乱 | 🟢 | ① | 命名唯一化（重复时追加序号） |
| 长连接被反向代理超时断开 | 🟡 | ② | 等待期持续保活 |
| 工作区文件冲突 | 🟢 | ② | 检测到冲突后给出回滚建议 |

### 11.3 开放问题（待后续迭代决策）

1. **完整事件日志**：当前以消息表持久化，是否升级为按事件序号存储的完整日志（支撑 time travel / 全量回放）？
2. **多 Agent 端实例**：当前 Agent 端单实例，是否扩展为多实例 + 负载均衡？
3. **双向通道时机**：当前坚持单向流式推送；何时进入「人机协同编辑」场景（用户与 Agent 同时改一段代码 / 文档）才需要引入双向通道？
4. **多租户 / 权限**：当前单人系统，产品化后是否引入用户体系与权限？
5. **更复杂的执行图**：Orchestrator 当前以「波次」串行 / 并行，是否升级为完整 DAG 调度？

### 11.4 风险纪律

守住「务实优先、不过度设计」：

- ❌ 不让后端变成第二个执行引擎（后端只中继与持久化，不执行 Agent）。
- ❌ 不让事件总线变成消息队列（不做持久化 / 确认 / 重试）。
- ❌ 不让产物系统变成数据 / 资产平台（不做版本 / 依赖 / 血缘）。
- ❌ 不让编排器变成通用调度框架（不接管订阅 / 状态机生命周期）。
