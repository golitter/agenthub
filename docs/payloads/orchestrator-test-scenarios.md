# Orchestrator 前端交互测试场景

> 用户输入自然语言，期待在聊天界面看到什么输出？按 Phase 5 设计文档验证。
> 日志位置：`logs/agentend.log`、`logs/backend.log`、`logs/frontend.log`

---

## 场景 1: 闲聊 — 直接回复

### 用户输入

> 你好，请介绍一下你自己

### 期待看到

**管理者（Orchestrator）的消息气泡**：直接回复一段文本，例如"你好！我是一个任务编排助手，可以协调多个 Agent 来完成任务...""。不会弹出任何卡片或触发子 Agent。

---

## 场景 2: 向子 Agent 提问

### 用户输入

> 这个项目能不能用 Docker 部署？帮我问一下执行者

### 期待看到

**先弹出一张 AskAgentCard 卡片**（展开态）：

```
┌─ Ask Card（pending）─────────────────────────────┐
│ [管理者] → [执行者]                   ●脉冲 等待回答│
│ ──────────────────────────────────────────────────│
│ 项目根目录有 Dockerfile 吗？Docker 部署情况如何？  │
└───────────────────────────────────────────────────┘
```

此时执行者的聊天框里会同步出现流式回答："有的，根目录有 Dockerfile，使用多阶段构建..."

**等执行者回答完毕**，AskAgentCard 自动折叠为一行：

```
┌─ Ask Card（collapsed）───────────────────────────┐
│ [管理者] → [执行者]  项目根目录有 Dockerfile 吗...  ● 已回答 │
└───────────────────────────────────────────────────┘
```

点击可再次展开。执行者的完整回答作为独立消息保留在聊天流中。

**最后管理者**会基于回答给出一个汇总文本："根据执行者的反馈，该项目已支持 Docker 部署，可以直接使用 docker build ..."

## 场景 3: 多 Agent 编排执行

### 用户输入

> 帮我实现一个登录页面，然后帮我审查一下代码

### 期待看到

**第一步 — 规划卡片**（PlanCard）出现在管理者聊天框：

```
┌──────────────────────────────────────────────────┐
│  拆解为 2 个子任务                                │
│                                                  │
│  ○ task-001  [执行者]  实现登录页面               │
│  ○ task-002  [审查员]  审查 task-001 代码          │
└──────────────────────────────────────────────────┘
```

**第二步 — 逐个执行**。每个任务的状态卡片（RuntimeStatus）出现在管理者聊天框：

```
  执行者 ●执行中
  ┌────────────────────────────────────────┐
  │ 正在创建 LoginPage.tsx...              │
  │ 正在添加表单验证...                     │
  │ ✓ 完成 (12.5s)                        │
  └────────────────────────────────────────┘

  审查员 ●执行中
  ┌────────────────────────────────────────┐
  │ 正在审查 LoginPage.tsx...              │
  │ 建议: 添加密码强度校验                  │
  │ ✓ 完成 (8.3s)                         │
  └────────────────────────────────────────┘
```

与此同时，执行者、审查员各自的聊天框中会显示完整的流式输出。

**第三步 — 汇总报告**。全部完成后，管理者输出一份 Aggregator 汇总报告，列出各任务的完成情况和关键产出。

## 场景 4: 任务失败 + 重规划

### 用户输入

> 帮我重构整个认证系统，改用 OAuth2.0

### 期待看到

执行者的任务卡片可能变成失败状态：

```
  执行者 ●失败
  ┌────────────────────────────────────────┐
  │ [Error] 当前项目没有认证模块，无法重构   │
  └────────────────────────────────────────┘
```

**管理者检测到失败后**，自动重新规划：PlanCard 更新为新方案（最多重试 3 次），然后重新 dispatch 执行。

---

## 场景 5: 协调通道（CoordChannel）

### 用户输入

> 帮我的前端项目加上完整的权限管理系统

### 期待看到

管理者不直接分配任务，而是先在聊天框里弹出一个**可折叠的协调通道面板**：

```
┌─ 协调通道（点击展开）──────────────────┐
│ 第 1 轮协调                            │
│ ┌────────────────────────────────────┐ │
│ │ 管理者 → 执行者:                      │ │
│ │ "项目目前用的什么技术栈？路由方案？"    │ │
│ ├────────────────────────────────────┤ │
│ │ 执行者 → 管理者:                      │ │
│ │ "React + React Router v6，用了 JWT"  │ │
│ ├────────────────────────────────────┤ │
│ │ 管理者 → 审查员:                      │ │
│ │ "审查权限时重点关注哪些安全点？"        │ │
│ ├────────────────────────────────────┤ │
│ │ 审查员 → 审查员:                      │ │
│ │ "重点关注 CSRF、XSS、token 刷新"      │ │
│ └────────────────────────────────────┘ │
│                                        │
│ 第 2 轮协调                            │
│  ...                                   │
│                                        │
│ ┌─ 协调结论 ──────────────────────────┐ │
│ │ 确定方案：基于 React Router 的路由     │ │
│ │ 守卫 + JWT token 刷新 + RBAC 权限模型 │ │
│ └─────────────────────────────────────┘ │
│                                        │
│ 2 轮协调，4 条消息                       │
└──────────────────────────────────────────┘
```

协调完成后，管理者正式输出 PlanCard 并开始 dispatch 执行。

## 日志排查方式

如果前端没有按预期渲染，依次检查：

```bash
# 1. 三端是否都正常运行
make status

# 2. AgentEnd 日志 — 看 orchestrator 各节点是否正常执行
tail -100 logs/agentend.log | grep -E "reason_node|ask_agent|dispatch|execute|ERROR"

# 3. Backend 日志 — 看 Redis Stream 是否正常转发
tail -100 logs/backend.log | grep -E "ERROR|Redis|agent service"

# 4. Frontend 日志 — 看 Vite HMR / 构建报错
tail -50 logs/frontend.log
```

**常见问题对照**：

| 现象 | 可能原因 | 查看日志 |
|------|---------|---------|
| Ask Card 一直"等待回答"不结束 | 子 Agent CLI 超时或 BackendClient 连接失败 | `agentend.log` 搜 `ask_agent` |
| PlanCard 不出现 | reason_node 未调用 plan_and_dispatch 或 LLM 返回格式不对 | `agentend.log` 搜 `plan` |
| RuntimeStatus 始终 pending | ExecutionEngine 未成功启动子 Agent 子进程 | `agentend.log` 搜 `ExecutionEngine` |
| 消息气泡被长文本撑开 | CSS overflow 问题（已修复：`min-w-0` + `break-words`） | 检查最近提交 |
| 协调通道不出现 | CoordChannel 功能已实现（`CoordChannel.tsx`），如果仍不出现检查 SSE 事件是否包含 `coordination_*` 类型 | `agentend.log` 搜 `coordination` |
